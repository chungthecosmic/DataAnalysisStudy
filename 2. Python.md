# Part 0. Python 기초 체크리스트

SQL을 많이 쓰다 보면 Python 기본 문법이 흐릿해지는 게 당연해요. 여기서부터 쌓고 갑시다.

## 0-1. 변수와 기본 타입

Python은 **타입 선언 없이** 변수를 만들어요. 할당하는 값에 따라 타입이 결정되죠.

```python
# 숫자
x = 10              # int
y = 3.14            # float

# 문자열 — 작은따옴표, 큰따옴표 아무거나 OK
name = "충현"
greeting = '안녕'

# 불리언 — True/False (첫 글자 대문자!)
is_active = True
is_deleted = False

# None — SQL의 NULL과 비슷
empty = None

# 타입 확인
type(x)             # <class 'int'>
type(name)          # <class 'str'>
```

**SQL과 다른 헷갈리는 포인트:**

|Python|SQL|
|---|---|
|`True`, `False` (대문자 첫글자)|`TRUE`, `FALSE`|
|`None`|`NULL`|
|`x == y` (동등 비교)|`x = y`|
|`x != y` (다름)|`x != y` 또는 `x <> y`|
|`5 / 2 = 2.5`|5 / 2 = 2.5 (DB에 따라 다름)|
|`5 // 2 = 2` (정수 나눗셈)|명시적 CAST 필요|
|`x and y`, `x or y`, `not x`|`AND`, `OR`, `NOT`|

## 0-2. 조건문 (if / elif / else)

```python
price = 50000

if price >= 100000:
    grade = "고가"
elif price >= 30000:
    grade = "중가"
else:
    grade = "저가"

# 한 줄로 쓰기 (삼항 연산자 비슷한 것)
grade = "고가" if price >= 100000 else "저가"
```

**중요한 Python 특징 — 들여쓰기가 문법이에요.**

```python
# ❌ 이렇게 하면 IndentationError
if price > 0:
grade = "valid"

# ✅ 반드시 들여쓰기 (보통 스페이스 4칸)
if price > 0:
    grade = "valid"
```

**복합 조건**

```python
# and, or, not
if price > 0 and price < 1000000:
    grade = "valid"

# in — 특정 값이 리스트/집합에 있는지
if category in ["디지털기기", "가구", "의류"]:
    print("주요 카테고리")

# is None — NULL 체크는 반드시 ==이 아니라 is 사용
if value is None:
    print("값 없음")

# 빈 값 판단 (falsy values: 0, "", [], {}, None)
items = []
if not items:   # 빈 리스트면 True
    print("비어있음")
```

## 0-3. for 반복문

```python
# 리스트 순회
for post in posts:
    print(post["category"])

# range() — 0부터 N-1까지
for i in range(5):           # 0, 1, 2, 3, 4
    print(i)

for i in range(1, 6):        # 1, 2, 3, 4, 5
    print(i)

for i in range(0, 10, 2):    # 0, 2, 4, 6, 8 (step=2)
    print(i)

# enumerate — 인덱스와 함께
for idx, post in enumerate(posts):
    print(f"{idx}번째: {post['category']}")

# zip — 두 리스트 같이 순회
ids = [1, 2, 3]
names = ["A", "B", "C"]
for i, n in zip(ids, names):
    print(i, n)

# break / continue
for p in posts:
    if p["price"] < 0:
        continue                 # 다음으로 넘김
    if p["price"] > 10000000:
        break                    # 완전 종료
    print(p)
```

## 0-5. 함수 정의

```python
def calculate_discount(price, rate=0.1):    # rate에 기본값
    """할인 적용 후 가격 반환 (문서화용 docstring)"""
    return price * (1 - rate)

calculate_discount(10000)              # 9000.0 (rate=0.1 기본값)
calculate_discount(10000, 0.2)         # 8000.0
calculate_discount(10000, rate=0.2)    # 8000.0 (키워드 인자)

# 여러 값 반환 → 튜플로 반환
def get_min_max(numbers):
    return min(numbers), max(numbers)

low, high = get_min_max([3, 1, 4, 1, 5])    # low=1, high=5
```

**자주 하는 실수 — mutable 기본 인자**

```python
# ❌ 위험! 함수 호출마다 공유됨
def add_item(item, items=[]):
    items.append(item)
    return items

# ✅ 올바른 방식
def add_item(item, items=None):
    if items is None:
        items = []
    items.append(item)
    return items
```

## 0-6. 문자열 포매팅 (f-string)

```python
name = "충현"
score = 95.5

# f-string (Python 3.6+ — 이거 하나만 기억하면 됨)
print(f"{name}님의 점수는 {score}점입니다")

# 포맷 지정
print(f"{score:.1f}")              # 95.5 (소수점 1자리)
print(f"{score:.0f}")              # 96 (반올림, 소수점 없음)
print(f"{12345:,}")                # 12,345 (천 단위 콤마)
print(f"{0.1234:.1%}")             # 12.3% (퍼센트)
print(f"{'hi':>10}")               # "        hi" (우측 정렬, 폭 10)
```

# Part 1. 핵심 자료형 복습

## 1-1. List (리스트)

**순서 있음, 중복 허용, 변경 가능**. SQL의 result set과 가장 비슷해요.

```python
# 생성
numbers = [1, 2, 3, 4, 5]
mixed = [1, "hello", True, None]       # 타입 섞어도 됨
empty = []

# 접근 (0부터 시작!)
numbers[0]       # 1
numbers[-1]      # 5 (마지막)
numbers[-2]      # 4 (뒤에서 두 번째)

# 슬라이싱 [start:end:step]  — end는 포함 안 됨
numbers[1:3]     # [2, 3]
numbers[:2]      # [1, 2]
numbers[2:]      # [3, 4, 5]
numbers[::-1]    # [5, 4, 3, 2, 1] (역순)
numbers[::2]     # [1, 3, 5] (2칸씩)

# 수정
numbers[0] = 100
numbers.append(6)              # [100, 2, 3, 4, 5, 6]
numbers.extend([7, 8])         # [100, 2, 3, 4, 5, 6, 7, 8]
numbers.insert(0, 999)         # 0번 자리에 999 삽입
numbers.remove(3)              # 값이 3인 첫 요소 제거
numbers.pop()                  # 마지막 요소 꺼내서 반환
numbers.pop(0)                 # 0번 요소 꺼내서 반환

# 유용한 메서드
len(numbers)                   # 길이
sum(numbers)                   # 합계
max(numbers), min(numbers)
sorted(numbers)                # 정렬된 새 리스트 반환
sorted(numbers, reverse=True)  # 내림차순
numbers.sort()                 # 원본을 정렬 (반환값 없음!)
3 in numbers                   # True/False
numbers.count(3)               # 3이 몇 번 나오는지
numbers.index(3)               # 3의 첫 위치
```

**append vs extend — 면접에서 자주 헷갈림**

```python
a = [1, 2]
a.append([3, 4])        # [1, 2, [3, 4]] — 통째로 하나의 요소로
a.extend([3, 4])        # [1, 2, 3, 4]   — 풀어서 추가
a + [3, 4]              # [1, 2, 3, 4]   — 새 리스트 반환
```

## 1-2. Dict (딕셔너리)

**key-value 저장, 중복 key 불가, Python 3.7+ 입력 순서 보장**. SQL의 row 하나를 표현하는 데 자주 써요.

```python
# 생성
post = {"post_id": 1, "category": "디지털기기", "price": 50000}
empty = {}

# 접근
post["category"]               # "디지털기기"
post.get("category")           # "디지털기기"
post.get("color")              # None (없는 key지만 에러 안 남)
post.get("color", "미지정")     # "미지정" (기본값)

# ⚠️ 차이 주의
post["color"]                  # KeyError 발생!
post.get("color")              # None 반환 (안전)

# 수정
post["price"] = 45000          # 기존 key 수정
post["new_key"] = "value"      # 새 key 추가
del post["new_key"]            # 삭제

# 순회
for key in post:               # key만
    print(key)

for key, value in post.items():    # key, value 둘 다
    print(key, value)

for value in post.values():    # value만
    print(value)

# 존재 확인
"category" in post             # True
"color" not in post            # True

# 유용한 메서드
post.keys()                    # dict_keys(['post_id', 'category', 'price'])
post.values()                  # dict_values([1, '디지털기기', 45000])
post.items()                   # dict_items([('post_id', 1), ...])
len(post)                      # 3

# 병합 (Python 3.9+)
a = {"x": 1, "y": 2}
b = {"y": 20, "z": 30}
c = a | b                      # {"x": 1, "y": 20, "z": 30}  (b가 우선)

# 또는 unpacking
c = {**a, **b}                 # 동일한 결과
```

## 1-3. Set (집합)

**중복 없음, 순서 없음, 수학적 집합 연산 지원**. SQL의 `DISTINCT`, `INTERSECT`, `EXCEPT`를 Python에서 쓰는 기분.

```python
# 생성
a = {1, 2, 3}
b = {2, 3, 4}
empty = set()                  # ⚠️ {} 는 빈 dict임!

# 리스트 → set (중복 제거)
unique = set([1, 1, 2, 2, 3])  # {1, 2, 3}

# 추가/제거
a.add(4)                       # {1, 2, 3, 4}
a.remove(4)                    # 없으면 KeyError
a.discard(999)                 # 없어도 에러 안 남

# 집합 연산 (SQL 비유)
a & b                          # {2, 3}        INTERSECT (교집합)
a | b                          # {1, 2, 3, 4}  UNION    (합집합)
a - b                          # {1}           EXCEPT   (차집합)
a ^ b                          # {1, 4}        대칭차집합 (어느 한쪽에만 있는 것)

# 포함 관계
{1, 2}.issubset({1, 2, 3})     # True (부분집합)
{1, 2, 3}.issuperset({1, 2})   # True (상위집합)

# 중요: 멤버십 테스트가 O(1)
999 in a                       # 매우 빠름 (리스트는 O(N))
```

**set vs list — 언제 뭘 써야 하나?**

|상황|선택|
|---|---|
|순서가 중요하다|list|
|중복 허용해야 한다|list|
|포함 여부 (`in`) 체크가 많다|set|
|두 그룹 비교 (교집합/차집합)|set|
|인덱스로 접근 (`[0]`)|list|

## 1-4. Tuple (튜플)

**순서 있음, 중복 허용, 변경 불가능(immutable)**. 고정된 짝을 표현할 때, 또는 dict key로 쓸 때.

```python
point = (3, 5)                 # x=3, y=5
point[0]                       # 3
# point[0] = 10                # TypeError! 변경 불가

# 괄호 생략 가능
t = 1, 2, 3                    # (1, 2, 3)

# 한 개짜리 튜플은 쉼표 필수
t = (1,)                       # 튜플
t = (1)                        # 그냥 정수 1

# unpacking
x, y = point                   # x=3, y=5
a, *rest = [1, 2, 3, 4]        # a=1, rest=[2, 3, 4]
a, b, c = "ABC"                # a='A', b='B', c='C'

# dict key로 사용 가능 (list는 안 됨!)
route_distances = {
    ("서울", "부산"): 400,
    ("서울", "대구"): 300,
}
```

# Part 2. 자주 쓰는 문법

## 2-1. lambda (익명 함수)

한 줄짜리 함수를 만들 때 써요. `sorted`, `map`, `filter` 같은 곳에서 정렬/변환 기준을 지정할 때 자주 나와요.

```python
# 일반 함수
def double(x):
    return x * 2

# lambda로 표현
double = lambda x: x * 2

# 더 복잡한 것도 가능
is_adult = lambda age: age >= 18
add = lambda a, b: a + b
```

**실전 예시 — sorted와 함께**

```python
posts = [
    {"id": 1, "price": 50000, "category": "A"},
    {"id": 2, "price": 30000, "category": "B"},
    {"id": 3, "price": 80000, "category": "A"},
]

# 가격 오름차순
sorted(posts, key=lambda p: p["price"])

# 가격 내림차순
sorted(posts, key=lambda p: p["price"], reverse=True)

# 카테고리 오름차순, 같으면 가격 내림차순
sorted(posts, key=lambda p: (p["category"], -p["price"]))
```

**max/min에서도 같은 패턴**

```python
max(posts, key=lambda p: p["price"])   # 가격 가장 높은 post
min(posts, key=lambda p: p["price"])   # 가격 가장 낮은 post
```

## 2-2. Comprehension (한 줄 변환)

`[...]`로 리스트를, `{...}`로 set 또는 dict를, `(...)`로 generator를 만들어요. SQL의 `SELECT ... WHERE`를 Python으로 옮기는 느낌.

```python
posts = [
    {"id": 1, "price": 50000, "category": "A"},
    {"id": 2, "price": 30000, "category": "B"},
    {"id": 3, "price": 80000, "category": "A"},
]

# list comprehension
prices = [p["price"] for p in posts]
# [50000, 30000, 80000]

# 조건 필터
expensive = [p for p in posts if p["price"] >= 50000]

# 변환 + 필터
tax_applied = [p["price"] * 1.1 for p in posts if p["category"] == "A"]

# if-else (표현식 위치 주의)
labels = ["비싸" if p["price"] >= 50000 else "싸" for p in posts]

# set comprehension — 중복 제거
categories = {p["category"] for p in posts}        # {'A', 'B'}

# dict comprehension
id_to_price = {p["id"]: p["price"] for p in posts}
# {1: 50000, 2: 30000, 3: 80000}

# generator expression — 메모리 절약
total = sum(p["price"] for p in posts)             # 리스트 안 만들고 바로 합
```

**일반 for문 vs comprehension**

```python
# 일반 for문
result = []
for p in posts:
    if p["price"] >= 50000:
        result.append(p["id"])

# comprehension (같은 결과)
result = [p["id"] for p in posts if p["price"] >= 50000]
```

Comprehension이 더 짧고 빠르지만, **로직이 복잡해지면 읽기 어려워져요**. 중첩 이중 루프까지는 OK, 그 이상은 일반 for문이 나아요.

## 2-3. Unpacking

여러 값을 한 번에 분해하거나 합치는 문법.

```python
# 기본
a, b = 1, 2                    # a=1, b=2
a, b = b, a                    # swap: a=2, b=1

# 리스트 unpacking
first, *rest = [1, 2, 3, 4]    # first=1, rest=[2, 3, 4]
*init, last = [1, 2, 3, 4]     # init=[1,2,3], last=4

# 함수 인자에 unpacking
def add(x, y, z):
    return x + y + z

args = [1, 2, 3]
add(*args)                     # add(1, 2, 3) = 6

kwargs = {"x": 1, "y": 2, "z": 3}
add(**kwargs)                  # add(x=1, y=2, z=3) = 6

# dict 병합
a = {"x": 1}
b = {"y": 2}
merged = {**a, **b, "z": 3}    # {"x": 1, "y": 2, "z": 3}
```

## 2-4. 자주 쓰는 내장 함수

```python
# len, sum, min, max
len([1, 2, 3])                 # 3
sum([1, 2, 3])                 # 6

# any / all
any([False, True, False])      # True (하나라도 True면)
all([True, True, True])        # True (모두 True면)

any(p["price"] > 100000 for p in posts)    # 하나라도 10만원 넘으면
all(p["price"] > 0 for p in posts)         # 모두 양수면

# sorted (+ reversed)
sorted([3, 1, 2])              # [1, 2, 3]
list(reversed([1, 2, 3]))      # [3, 2, 1]

# map / filter (comprehension으로 대체 가능, 읽을 줄은 알아야 함)
list(map(lambda x: x * 2, [1, 2, 3]))       # [2, 4, 6]
list(filter(lambda x: x > 1, [1, 2, 3]))    # [2, 3]
```

# Part 3. 실전 — pandas로 데이터 다루기

여기부터 실전이에요. pandas는 SQL의 테이블과 비슷한 `DataFrame`이라는 자료구조를 다루는 라이브러리예요. **Data Analytics Engineer가 Python으로 데이터를 다룬다면 99% pandas를 씁니다.**

## 3-1. pandas 기초 — SQL과 매핑해서 이해하기

```python
import pandas as pd

# DataFrame 생성
df = pd.DataFrame([
    {"user_id": 1, "category": "디지털기기", "price": 50000, "ts": "2026-04-18"},
    {"user_id": 2, "category": "가구",       "price": 30000, "ts": "2026-04-18"},
    {"user_id": 1, "category": "디지털기기", "price": 80000, "ts": "2026-04-19"},
    {"user_id": 3, "category": "의류",       "price": 15000, "ts": "2026-04-19"},
    {"user_id": 1, "category": "가구",       "price": 120000, "ts": "2026-04-20"},
])

# 기본 탐색
df.head()                      # 앞 5줄
df.tail(3)                     # 뒤 3줄
df.shape                       # (5, 4)
df.columns                     # Index(['user_id', 'category', ...])
df.dtypes                      # 각 컬럼 타입
df.info()                      # 요약 정보
df.describe()                  # 숫자 컬럼 통계
```

**SQL ↔ pandas 치환표**

|SQL|pandas|
|---|---|
|`SELECT col1, col2 FROM t`|`df[["col1", "col2"]]`|
|`SELECT * FROM t WHERE x > 5`|`df[df["x"] > 5]`|
|`WHERE x > 5 AND y = 'A'`|`df[(df["x"] > 5) & (df["y"] == "A")]`|
|`ORDER BY col DESC`|`df.sort_values("col", ascending=False)`|
|`LIMIT 10`|`df.head(10)`|
|`SELECT DISTINCT col`|`df["col"].unique()` 또는 `df.drop_duplicates()`|
|`COUNT(*)`|`len(df)` 또는 `df.shape[0]`|
|`GROUP BY col`|`df.groupby("col")`|
|`JOIN`|`df1.merge(df2, on="key")`|
|`UNION ALL`|`pd.concat([df1, df2])`|
|`CASE WHEN`|`df["col"].apply(lambda x: ...)` 또는 `np.where`|
|`IS NULL`|`df["col"].isna()`|
|`COALESCE`|`df["col"].fillna(value)`|
|`ROW_NUMBER OVER`|`df.groupby("g").cumcount()` 또는 `rank()`|

### 문제 4-1-1: Base 모델 — 타입 정규화와 컬럼 리네이밍

> 원천 데이터의 컬럼 이름/타입을 표준 규격으로 정제하는 함수를 작성하세요.

```python
import pandas as pd

def transform_base_users(raw: pd.DataFrame) -> pd.DataFrame:
    """
    원천 users 테이블을 base 레이어 규격으로 변환.

    규격:
    - id → user_id (string)
    - userName → user_name
    - createdAt → created_at (datetime)
    - isLeader → is_leader (bool)
    - status_code → user_status (string, 매핑)
    """
    status_map = {
        1: "NEW",
        2: "REACTIVATED",
        3: "ACTIVATED",
    }

    df = raw.copy()
    df = df.rename(columns={
        "id": "user_id",
        "userName": "user_name",
        "createdAt": "created_at",
        "isLeader": "is_leader",
    })

    # 타입 변환
    df["user_id"] = df["user_id"].astype(str)
    df["created_at"] = pd.to_datetime(df["created_at"])
    df["is_leader"] = df["is_leader"].astype(bool)

    # status 매핑
    df["user_status"] = df["status_code"].map(status_map).fillna("DEFAULT")

    # 필요한 컬럼만
    return df[["user_id", "user_name", "created_at", "is_leader", "user_status"]]


# 테스트
raw = pd.DataFrame([
    {"id": 1, "userName": "Alice", "createdAt": "2026-01-01",
     "isLeader": 1, "status_code": 1},
    {"id": 2, "userName": "Bob",   "createdAt": "2026-02-01",
     "isLeader": 0, "status_code": 3},
    {"id": 3, "userName": "Carol", "createdAt": "2026-03-01",
     "isLeader": 0, "status_code": 99},     # 매핑 없는 코드
])
```

### 문제 4-1-2: Fact 모델 — "유저가 메시지를 보냈다" 행동 단위 변환

> 원천 이벤트 로그에서 "메시지 전송" 이벤트만 추출하고, 메시지 카테고리를 부여하세요.

```python
def transform_fact_user_sent_message(
    events: pd.DataFrame,
    target_date: str,
) -> pd.DataFrame:
    """
    target_date 하루치 이벤트에서 'message_sent' 행동을 추출.

    입력: events (columns: event_id, user_id, event_type, payload, event_timestamp)
    출력: user_id, message_category, event_timestamp 등
    """
    df = events.copy()
    df["event_timestamp"] = pd.to_datetime(df["event_timestamp"])

    # 파티션 필터 — dbt incremental + insert_overwrite의 WHERE 절과 같은 역할
    mask = df["event_timestamp"].dt.date == pd.to_datetime(target_date).date()
    df = df[mask]

    # 메시지 전송 이벤트만
    df = df[df["event_type"] == "message_sent"]

    # payload (dict)에서 category 추출 — 없으면 'unknown'
    df["message_category"] = df["payload"].apply(
        lambda p: p.get("category", "unknown") if isinstance(p, dict) else "unknown"
    )

    # 결과 스키마
    result = df[["event_id", "user_id", "message_category", "event_timestamp"]]

    # meta 컬럼 (당근 Fact 모델 규약에 있음)
    result = result.assign(
        created_at=pd.Timestamp.now(),
        updated_at=pd.Timestamp.now(),
    )
    return result


# 테스트
```

## 4-2. 데이터 품질 체크 함수 (dbt test의 Python 버전)

dbt test (not_null, unique, accepted_values 등)에 해당하는 검증 함수를 Python으로 구현하는 스타일.

### 문제 4-2-1: 기본 DQ 체크 함수

> 주어진 DataFrame에 대해 컬럼 단위 품질 체크를 수행하는 함수를 작성하세요.

```python
from typing import List, Dict, Any


def check_not_null(df: pd.DataFrame, column: str) -> Dict[str, Any]:
    """컬럼에 NULL이 없는지 검증."""
    null_count = df[column].isna().sum()
    return {
        "check": "not_null",
        "column": column,
        "passed": null_count == 0,
        "null_count": int(null_count),
        "total_rows": len(df),
    }


def check_unique(df: pd.DataFrame, column: str) -> Dict[str, Any]:
    """컬럼 값이 고유한지 검증."""
    duplicate_count = df[column].duplicated().sum()
    return {
        "check": "unique",
        "column": column,
        "passed": duplicate_count == 0,
        "duplicate_count": int(duplicate_count),
        "total_rows": len(df),
    }


def check_accepted_values(
    df: pd.DataFrame,
    column: str,
    accepted: List[Any],
) -> Dict[str, Any]:
    """컬럼 값이 허용된 목록 안에 있는지 검증."""
    invalid_mask = ~df[column].isin(accepted)
    invalid_count = invalid_mask.sum()
    return {
        "check": "accepted_values",
        "column": column,
        "passed": invalid_count == 0,
        "invalid_count": int(invalid_count),
        "invalid_values": df.loc[invalid_mask, column].unique().tolist(),
    }


def run_dq_checks(df: pd.DataFrame, checks: List[Dict]) -> List[Dict]:
    """
    검증 정의 리스트를 받아 각각 실행하고 결과 리스트 반환.

    checks 예시:
    [
        {"type": "not_null", "column": "user_id"},
        {"type": "unique",   "column": "event_id"},
        {"type": "accepted_values", "column": "status",
         "values": ["NEW", "ACTIVATED"]},
    ]
    """
    results = []
    for chk in checks:
        if chk["type"] == "not_null":
            results.append(check_not_null(df, chk["column"]))
        elif chk["type"] == "unique":
            results.append(check_unique(df, chk["column"]))
        elif chk["type"] == "accepted_values":
            results.append(check_accepted_values(df, chk["column"], chk["values"]))
        else:
            raise ValueError(f"Unknown check type: {chk['type']}")
    return results


# 테스트
df = pd.DataFrame([
    {"event_id": "e1", "user_id": 1, "status": "NEW"},
    {"event_id": "e2", "user_id": 2, "status": "ACTIVATED"},
    {"event_id": "e1", "user_id": None, "status": "UNKNOWN"},   # 중복 + NULL + 잘못된 status
])

checks = [
    {"type": "not_null", "column": "user_id"},
    {"type": "unique",   "column": "event_id"},
    {"type": "accepted_values", "column": "status",
     "values": ["NEW", "ACTIVATED", "REACTIVATED"]},
]

results = run_dq_checks(df, checks)
for r in results:
    print(r)

# 하나라도 실패하면 전체 실패로 처리 (Airflow task 실패 트리거)
all_passed = all(r["passed"] for r in results)
print(f"\nAll checks passed: {all_passed}")
```

**포인트**

- dbt test의 `not_null`, `unique`, `accepted_values`를 함수로 구현
- 각 체크가 **일관된 결과 포맷**(dict)으로 반환 → 나중에 로깅/알림 쉽게
- `run_dq_checks`는 "여러 체크 한 번에" — 실제 Airflow task에서 이런 패턴이에요
- `all(...)`로 전체 pass 여부 결정 → 실패 시 task fail

## 4-3. Airflow PythonOperator용 함수

Airflow의 `PythonOperator`는 **일반 Python 함수를 task로 실행**해줘요. 특징은 함수 시그니처에 `**context`를 받고, Airflow의 `execution_date` 같은 정보를 쓴다는 점.

### 문제 4-3-1: 일일 집계 task

> Airflow PythonOperator로 실행되어, 해당 execution_date의 events를 집계해 결과를 반환하는 함수를 작성하세요.

```python
from datetime import datetime, timedelta


def daily_aggregation_task(**context) -> Dict[str, Any]:
    """
    Airflow PythonOperator에서 실행되는 일일 집계 task.

    context에서 execution_date 꺼내서, 해당 날짜 이벤트만 집계.
    """
    # Airflow가 주입해주는 execution_date
    execution_date = context["execution_date"]
    target_date = execution_date.date()

    # 실제로는 BigQuery에서 읽어옴 (여기선 mock)
    events = read_events_from_source(target_date)

    # 집계
    summary = events.groupby("action").agg(
        event_count=("event_id", "count"),
        unique_users=("user_id", "nunique"),
    ).reset_index()

    # 결과 적재 (실제로는 BigQuery로 쓰기)
    write_to_target(summary, target_date)

    # Airflow가 XCom으로 공유할 메타데이터 반환
    return {
        "target_date": str(target_date),
        "row_count": len(summary),
        "actions_found": summary["action"].tolist(),
    }
```

### 문제 4-3-2: 백필 task — 날짜 범위 받아서 반복 실행

> 시작일과 종료일을 받아, 그 사이 각 날짜별로 집계를 수행하는 백필 함수를 작성하세요.

```python
def backfill_task(start_date: str, end_date: str, **context) -> List[Dict]:
    """
    start_date부터 end_date까지 각 날짜별로 집계 수행.

    당근 dbt/Airflow 글에서 소개된 백필 DAG의 Python 버전이에요.
    """
    start = pd.to_datetime(start_date).date()
    end = pd.to_datetime(end_date).date()

    results = []
    current = start
    while current <= end:
        try:
            events = read_events_from_source(current)

            summary = events.groupby("action").agg(
                event_count=("event_id", "count"),
                unique_users=("user_id", "nunique"),
            ).reset_index()

            write_to_target(summary, current)

            results.append({
                "date": str(current),
                "status": "success",
                "row_count": len(summary),
            })
        except Exception as e:
            # 한 날짜 실패해도 다음 날짜 계속 진행
            results.append({
                "date": str(current),
                "status": "failed",
                "error": str(e),
            })

        current += timedelta(days=1)

    # 실패 개수가 0이 아니면 task fail 유도
    failed = [r for r in results if r["status"] == "failed"]
    if failed:
        print(f"WARNING: {len(failed)} dates failed")

    return results
```

**포인트**

- 날짜 루프 + 각 날짜별 독립 처리
- 한 날짜 실패가 다음 날짜를 막지 않도록 try/except
- 결과 수집 → 마지막에 요약 → 로깅/알림
- 실제 Airflow에서는 `dynamic task mapping`을 쓰면 더 나음 (당근 글에 나옴)

## 4-4. CDC 스타일 — UPSERT 로직을 Python으로

당근 MongoDB CDC 글처럼, 변경 이벤트를 처리해 최종 상태를 만드는 패턴.

### 문제 4-4-1: Change Event를 snapshot으로 머지

> Change event 리스트가 주어졌을 때 primary key별로 최신 상태만 남겨 최종 테이블을 만드세요.

```python
def merge_cdc_events(
    current_snapshot: pd.DataFrame,
    change_events: pd.DataFrame,
    primary_key: str = "user_id",
) -> pd.DataFrame:
    """
    현재 snapshot + CDC 변경 이벤트 → 최종 snapshot 생성.

    - change_events의 operation_type: 'insert' / 'update' / 'delete'
    - 같은 key 여러 이벤트면 ts_ms가 가장 큰 것만 사용
    """
    # 같은 key에 여러 이벤트가 있으면 가장 최근만
    latest = change_events.sort_values("ts_ms").drop_duplicates(
        subset=[primary_key], keep="last"
    )

    # delete는 최종 결과에서 제외
    deleted_keys = set(latest.loc[latest["operation_type"] == "delete", primary_key])
    upserted = latest[latest["operation_type"].isin(["insert", "update"])]

    # 기존 snapshot에서 변경/삭제된 key 제거
    affected_keys = set(latest[primary_key])
    unchanged = current_snapshot[~current_snapshot[primary_key].isin(affected_keys)]

    # upsert된 이벤트에서 snapshot 컬럼만 뽑기
    snapshot_cols = current_snapshot.columns.tolist()
    upsert_rows = upserted[[c for c in snapshot_cols if c in upserted.columns]]

    # 합치기
    result = pd.concat([unchanged, upsert_rows], ignore_index=True)
    return result


# 테스트
current = pd.DataFrame([
    {"user_id": 1, "name": "Alice", "status": "active"},
    {"user_id": 2, "name": "Bob",   "status": "active"},
    {"user_id": 3, "name": "Carol", "status": "active"},
])

changes = pd.DataFrame([
    {"user_id": 2, "name": "Bob",    "status": "inactive",
     "operation_type": "update", "ts_ms": 100},
    {"user_id": 3, "name": "Carol",  "status": "active",
     "operation_type": "delete", "ts_ms": 150},
    {"user_id": 4, "name": "David",  "status": "active",
     "operation_type": "insert", "ts_ms": 200},
    {"user_id": 2, "name": "Bob2",   "status": "active",
     "operation_type": "update", "ts_ms": 300},    # 같은 user_id 두 번째 이벤트
])

print(merge_cdc_events(current, changes))
```

**포인트**

- 같은 key에 여러 이벤트면 **가장 늦은 타임스탬프만** (당근 CDC 글의 "fullDocument" 접근과 같은 아이디어)
- delete는 결과에서 제외
- 기존 snapshot에서 영향받은 key만 치환, 나머지는 그대로 (incremental + insert_overwrite 스타일)

## 5-1. 받은 문제를 풀 때 순서

1. **문제 이해 확인** — "이 데이터에 중복이 있을 수 있나요?", "NULL은 어떻게 처리할까요?"
2. **접근 설명** — "pandas로 groupby한 뒤 merge로 조인하겠습니다" 정도로 먼저 말
3. **간단한 것부터** — 먼저 돌아가는 코드, 그 다음 엣지케이스
4. **변수명 의미 있게** — `tmp`, `d`, `x` 피하고 `daily_summary`, `user_events`
5. **중간 결과 확인** — `print(df.head())`로 확인하면서 진행
6. **엣지케이스 언급** — "빈 DataFrame이 들어올 경우를 위해 early return 추가하겠습니다"

## 5-2. pandas로 풀 때 자주 막히는 지점

- **조건 필터에서 `and`/`or` 쓰기** → `&`/`|`와 괄호 필수
- **NaN 비교를 `== np.nan`으로** → `.isna()` 써야 함
- **chained indexing** → `.loc`로 명시
- **`groupby` 후 reset_index 까먹기** → index가 컬럼으로 안 나옴
- **datetime 변환 안 하고 연산** → `pd.to_datetime()` 먼저
- **SettingWithCopyWarning** → `.copy()`로 명시적 복사

## 5-3. 파이프라인 코드 짤 때 의식할 것

- **순수 함수 우선** — 입력 → 출력만, side effect 최소화
- **Idempotent** — 같은 입력에 같은 결과 (target_date를 받는 이유)
- **Early validation** — 시작 시 필수 컬럼/타입 체크
- **에러는 명확하게** — `ValueError("column X missing")` 수준
- **반환값은 구조화** — dict로 일관된 포맷

## 5-4. 면접관이 좋아하는 한 마디

- "SQL로 쓰면 `ROW_NUMBER OVER (PARTITION BY ...)` 인데, pandas로는 `groupby().cumcount()` 입니다"
- "데이터가 커지면 이 `apply` 부분이 bottleneck이 될 수 있어서, 벡터 연산으로 바꾸는 게 낫겠습니다"
- "이 함수는 dbt Fact 모델의 변환 로직을 Python으로 옮긴 구조이고, target_date를 파라미터로 받아 idempotent하게 만들었습니다"
- "실제 dbt test의 not_null과 동일한 검증인데, 결과를 Airflow XCom으로 넘기도록 dict 포맷으로 반환했습니다"
- "빈 입력이 들어왔을 때 KeyError 대신 빈 DataFrame을 반환하도록 가드 추가하겠습니다"

## 5-5. 마지막으로 — 관점

**"쉽게 쓸 수 있게"** 예요. 단순히 돌아가는 코드보다 **"다른 사람이 3개월 뒤에 봐도 이해할 수 있는 코드"** 가 더 좋은 평가를 받아요.

- 변수명, 함수명이 **자기 문서화(self-documenting)** 되어 있는지
- 함수의 **책임이 단일한지** (한 함수가 너무 많은 일 하지 않는지)
- **엣지케이스**를 의식적으로 챙겼는지
- 완벽하진 않아도 **어디까지 고민했는지** 소리내서 설명할 수 있는지

이 네 가지가 결국 **"내가 당근에서 같이 일하고 싶은 사람인가"** 의 판단 기준이 됩니다.
