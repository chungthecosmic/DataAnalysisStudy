# 당근 Data Analytics Engineer — SQL 코딩 테스트 준비 가이드

> 모든 예시는 당근 서비스(중고거래, 동네생활, 당근페이, 알바 등)를 가정한 테이블 구조를 사용합니다. BigQuery 문법 기준이지만, 대부분의 쿼리는 Redshift/PostgreSQL에서도 동일하게 동작합니다.

## 공통 테이블 정의

아래 테이블들을 모든 문제에서 공통으로 사용합니다.

```sql
-- 유저 테이블
CREATE TABLE users (
    user_id       BIGINT,
    signup_date   DATE,
    region        VARCHAR(50),   -- 동네 (예: '서초동', '역삼동')
    platform      VARCHAR(10)    -- 'ios' / 'android'
);

-- 유저 활동 로그 (앱 접속 기록)
CREATE TABLE user_activity (
    user_id       BIGINT,
    activity_date DATE,
    activity_type VARCHAR(30)    -- 'app_open', 'search', 'view_post', 'chat', 'write_post' 등
);

-- 중고거래 게시글
CREATE TABLE posts (
    post_id       BIGINT,
    user_id       BIGINT,        -- 판매자
    category      VARCHAR(30),   -- '디지털기기', '가구', '의류' 등
    price         INT,
    created_at    TIMESTAMP,
    status        VARCHAR(20)    -- 'active', 'reserved', 'sold', 'deleted'
);

-- 채팅 메시지
CREATE TABLE chats (
    chat_id       BIGINT,
    post_id       BIGINT,
    buyer_id      BIGINT,
    seller_id     BIGINT,
    first_msg_at  TIMESTAMP
);

-- 거래 완료
CREATE TABLE transactions (
    transaction_id BIGINT,
    post_id        BIGINT,
    buyer_id       BIGINT,
    seller_id      BIGINT,
    amount         INT,
    completed_at   TIMESTAMP
);
```

## 1. 잔존율 (Retention) 계산

### 왜 중요한가?

당근 같은 플랫폼 서비스에서 가장 핵심적인 지표입니다. 면접에서 나올 확률이 가장 높아요. 리텐션은 "가입한 유저가 이후에도 계속 서비스를 쓰는가"를 측정합니다.

### 문제 1-1: Classic Retention (D1, D7, D30)

> 가입 주차(cohort)별로 D1, D7, D30 리텐션을 구하세요. 리텐션 = 해당 일차에 재방문한 유저 수 / 코호트 유저 수

```sql
WITH cohort AS (
    SELECT
        user_id,
        signup_date,
        DATE_TRUNC('week', signup_date) AS cohort_week
    FROM users
),

activity AS (
    SELECT DISTINCT
        user_id,
        activity_date
    FROM user_activity
)

SELECT
    c.cohort_week,
    COUNT(DISTINCT c.user_id) AS cohort_size,

    -- D1: 가입 다음날 재방문
    COUNT(DISTINCT CASE
        WHEN DATE_DIFF(a.activity_date, c.signup_date, DAY) = 1
        THEN c.user_id
    END) AS d1_retained,

    -- D7: 가입 7일 후 재방문
    COUNT(DISTINCT CASE
        WHEN DATE_DIFF(a.activity_date, c.signup_date, DAY) = 7
        THEN c.user_id
    END) AS d7_retained,

    -- D30: 가입 30일 후 재방문
    COUNT(DISTINCT CASE
        WHEN DATE_DIFF(a.activity_date, c.signup_date, DAY) = 30
        THEN c.user_id
    END) AS d30_retained,

    -- 비율 계산
    ROUND(COUNT(DISTINCT CASE
        WHEN DATE_DIFF(a.activity_date, c.signup_date, DAY) = 1
        THEN c.user_id END) * 100.0 / COUNT(DISTINCT c.user_id), 2) AS d1_rate,

    ROUND(COUNT(DISTINCT CASE
        WHEN DATE_DIFF(a.activity_date, c.signup_date, DAY) = 7
        THEN c.user_id END) * 100.0 / COUNT(DISTINCT c.user_id), 2) AS d7_rate,

    ROUND(COUNT(DISTINCT CASE
        WHEN DATE_DIFF(a.activity_date, c.signup_date, DAY) = 30
        THEN c.user_id END) * 100.0 / COUNT(DISTINCT c.user_id), 2) AS d30_rate

FROM cohort c
LEFT JOIN activity a ON c.user_id = a.user_id
GROUP BY c.cohort_week
ORDER BY c.cohort_week;
```

**핵심 포인트:**

- `LEFT JOIN`을 써야 활동이 없는 유저도 cohort_size에 포함됨
- `COUNT(DISTINCT ...)` 안에 `CASE WHEN`을 넣는 패턴은 매우 자주 쓰임
- `DATE_DIFF` = 정확히 N일차 (BigQuery) / `DATEDIFF(day, ...)` (Redshift)

### 문제 1-2: Rolling Retention (범위 기반)

> D1~D7 사이에 한 번이라도 접속한 유저 비율을 구하세요.

```sql
WITH cohort AS (
    SELECT user_id, signup_date
    FROM users
    WHERE signup_date BETWEEN '2026-01-01' AND '2026-01-31'
),

revisit AS (
    SELECT DISTINCT c.user_id
    FROM cohort c
    INNER JOIN user_activity a ON c.user_id = a.user_id
    WHERE DATE_DIFF(a.activity_date, c.signup_date, DAY) BETWEEN 1 AND 7
)

SELECT
    COUNT(DISTINCT r.user_id) AS retained_users,
    COUNT(DISTINCT c.user_id) AS total_users,
    ROUND(COUNT(DISTINCT r.user_id) * 100.0
        / COUNT(DISTINCT c.user_id), 2) AS rolling_d7_rate
FROM cohort c
LEFT JOIN revisit r ON c.user_id = r.user_id;
```

**Classic vs Rolling 차이:**

- Classic: 정확히 D7에 접속했는가 → 엄격한 기준
- Rolling: D1~D7 사이 아무 때나 접속했는가 → 관대한 기준
- 면접에서 "어떤 리텐션 정의를 쓸 건가요?"라고 물어볼 수 있으니 차이를 설명할 수 있어야 함

## 2. Window Function 활용

### 문제 2-1: 유저별 첫 거래와 두 번째 거래 사이의 간격

> 거래를 2회 이상 한 판매자에 대해, 첫 번째와 두 번째 거래 완료 사이의 평균 일수를 구하세요.

```sql
WITH ranked_txn AS (
    SELECT
        seller_id,
        completed_at,
        ROW_NUMBER() OVER (
            PARTITION BY seller_id
            ORDER BY completed_at
        ) AS txn_seq
    FROM transactions
)

SELECT
    ROUND(AVG(DATE_DIFF(t2.completed_at, t1.completed_at, DAY)), 1) AS avg_days_to_2nd_txn
FROM ranked_txn t1
INNER JOIN ranked_txn t2
    ON t1.seller_id = t2.seller_id
    AND t1.txn_seq = 1
    AND t2.txn_seq = 2;
```

**핵심 포인트:**

- `ROW_NUMBER()`로 순서를 매긴 후 Self Join하는 패턴
- `PARTITION BY`는 그룹 단위, `ORDER BY`는 정렬 기준 — 이 두 가지를 명확히 구분해서 설명할 것

### 문제 2-2: LAG/LEAD로 이전/다음 활동 비교

> 유저별로 앱 접속 간격(일)의 평균을 구하세요. 7일 이상 접속 간격이 벌어진 유저를 이탈 위험 유저로 분류하세요.

```sql
WITH activity_gap AS (
    SELECT
        user_id,
        activity_date,
        LAG(activity_date) OVER (
            PARTITION BY user_id
            ORDER BY activity_date
        ) AS prev_activity_date,
        DATE_DIFF(
            activity_date,
            LAG(activity_date) OVER (
                PARTITION BY user_id
                ORDER BY activity_date
            ),
            DAY
        ) AS days_since_last
    FROM (
        SELECT DISTINCT user_id, activity_date
        FROM user_activity
    )
)

SELECT
    user_id,
    ROUND(AVG(days_since_last), 1) AS avg_gap_days,
    MAX(days_since_last) AS max_gap_days,
    CASE
        WHEN MAX(days_since_last) >= 7 THEN 'churn_risk'
        ELSE 'active'
    END AS user_status
FROM activity_gap
WHERE days_since_last IS NOT NULL   -- 첫 활동은 LAG가 NULL
GROUP BY user_id;
```

**핵심 포인트:**

- `LAG()` = 이전 행, `LEAD()` = 다음 행
- Window Function은 WHERE 절보다 나중에 실행되므로, NULL 필터는 바깥 쿼리에서 해야 함
- 서브쿼리에서 `DISTINCT`로 중복 날짜를 먼저 제거하는 것이 중요

### 문제 2-3: RANK / DENSE_RANK로 카테고리별 Top-N

> 카테고리별 거래 완료 건수 Top 3 판매자를 구하세요.

```sql
WITH seller_stats AS (
    SELECT
        p.category,
        t.seller_id,
        COUNT(*) AS txn_count
    FROM transactions t
    INNER JOIN posts p ON t.post_id = p.post_id
    GROUP BY p.category, t.seller_id
),

ranked AS (
    SELECT
        *,
        DENSE_RANK() OVER (
            PARTITION BY category
            ORDER BY txn_count DESC
        ) AS rnk
    FROM seller_stats
)

SELECT category, seller_id, txn_count, rnk
FROM ranked
WHERE rnk <= 3
ORDER BY category, rnk;
```

**ROW_NUMBER vs RANK vs DENSE_RANK 차이:**

|함수|동점 처리|결과 예시 (값: 100, 100, 90)|
|---|---|---|
|`ROW_NUMBER`|동점이어도 순번 다름|1, 2, 3|
|`RANK`|동점은 같은 순위, 다음 건너뜀|1, 1, 3|
|`DENSE_RANK`|동점은 같은 순위, 다음 안 건너뜀|1, 1, 2|

- Top-N 문제에서 동점자를 모두 포함하려면 `DENSE_RANK`, 정확히 N명만 뽑으려면 `ROW_NUMBER`

### 문제 2-4: NTILE / PERCENT_RANK — 분위 기반 세그먼트

> 거래 금액 기준으로 구매자를 5개 그룹(quintile)으로 나누고, 각 그룹의 평균 거래액과 인원 수를 구하세요.

```sql
WITH buyer_spend AS (
    SELECT
        buyer_id,
        SUM(amount) AS total_spend
    FROM transactions
    WHERE completed_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
    GROUP BY buyer_id
),

quintiled AS (
    SELECT
        buyer_id,
        total_spend,
        NTILE(5) OVER (ORDER BY total_spend DESC) AS spend_quintile,
        PERCENT_RANK() OVER (ORDER BY total_spend) AS pct_rank
    FROM buyer_spend
)

SELECT
    spend_quintile,
    COUNT(*) AS buyer_count,
    ROUND(AVG(total_spend), 0) AS avg_spend,
    MIN(total_spend) AS min_spend,
    MAX(total_spend) AS max_spend
FROM quintiled
GROUP BY spend_quintile
ORDER BY spend_quintile;
```

**잘 안 쓰지만 알아둬야 할 함수들:**

|함수|의미|예시|
|---|---|---|
|`NTILE(N)`|데이터를 N개의 균등 그룹으로 분할|상위 20% / 하위 20% 구분|
|`PERCENT_RANK()`|백분위 (0.0 ~ 1.0)|0.95 = 상위 5%|
|`CUME_DIST()`|누적 분포 (이 값 이하 비율)|RFM 분석에 활용|

- 면접 팁: "VIP 유저를 어떻게 정의할지" 물어볼 때 `NTILE`이나 `PERCENT_RANK`로 백분위 기반 정의를 제시하면 좋음

### 문제 2-5: 누적합 (Running Sum)

> 일별 신규 게시글 수와 누적 게시글 수를 구하세요.

```sql
SELECT
    DATE(created_at) AS post_date,
    COUNT(*) AS daily_posts,
    SUM(COUNT(*)) OVER (
        ORDER BY DATE(created_at)
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS cumulative_posts
FROM posts
GROUP BY DATE(created_at)
ORDER BY post_date;
```

**핵심 포인트:**

- `SUM() OVER (ORDER BY ...)` = 누적합 (Running Total)
- `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`가 기본값이지만 명시하는 게 좋음
- 집계함수 안에 집계함수를 넣을 수 있음 (`SUM(COUNT(*)) OVER (...)`)

### 문제 2-6: 윈도우 프레임 — 7일 이동 평균

> 일별 신규 가입자 수의 7일 이동 평균을 구하세요. (지난 6일 + 당일 = 7일)

```sql
WITH daily_signups AS (
    SELECT
        signup_date,
        COUNT(*) AS new_users
    FROM users
    GROUP BY signup_date
)

SELECT
    signup_date,
    new_users,
    -- 행 기준: 지난 6행 + 현재행
    ROUND(AVG(new_users) OVER (
        ORDER BY signup_date
        ROWS BETWEEN 6 PRECEDING AND CURRENT ROW
    ), 1) AS ma7_rows,

    -- 날짜 기준: 6일 전 ~ 오늘 (데이터 빈 날 있어도 정확)
    ROUND(AVG(new_users) OVER (
        ORDER BY UNIX_DATE(signup_date)  -- BigQuery
        RANGE BETWEEN 6 PRECEDING AND CURRENT ROW
    ), 1) AS ma7_range
FROM daily_signups
ORDER BY signup_date;
```

**ROWS vs RANGE — 자주 헷갈리는 포인트:**

|모드|기준|빈 날짜 처리|
|---|---|---|
|`ROWS BETWEEN N PRECEDING AND CURRENT ROW`|행 개수|빈 날짜 무시하고 N개 행만 사용 → **잘못된 평균 가능**|
|`RANGE BETWEEN N PRECEDING AND CURRENT ROW`|값(날짜) 범위|실제 날짜 범위 기준 → **빈 날짜도 고려됨**|

- 시계열 데이터에서 일자가 비어있을 가능성이 있다면 `RANGE` 사용 권장
- 면접 팁: "7일 이동평균"이라는 요구가 들어왔을 때 ROWS/RANGE 차이를 짚어주면 가산점

### 문제 2-7: FIRST_VALUE / LAST_VALUE / NTH_VALUE

> 각 유저의 모든 거래 기록과 함께, 그 유저의 첫 거래일과 가장 최근 거래일을 같이 출력하세요.

```sql
SELECT
    buyer_id,
    completed_at,
    amount,
    FIRST_VALUE(completed_at) OVER (
        PARTITION BY buyer_id
        ORDER BY completed_at
    ) AS first_purchase_at,

    -- ⚠️ LAST_VALUE는 기본 프레임이 'CURRENT ROW까지'라서 실제 마지막 값이 안 나옴
    LAST_VALUE(completed_at) OVER (
        PARTITION BY buyer_id
        ORDER BY completed_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS last_purchase_at,

    NTH_VALUE(completed_at, 2) OVER (
        PARTITION BY buyer_id
        ORDER BY completed_at
        ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING
    ) AS second_purchase_at
FROM transactions
ORDER BY buyer_id, completed_at;
```

**숨은 함정 — `LAST_VALUE` 프레임:**

- Window Function의 기본 프레임은 `RANGE BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW`
- 즉, `LAST_VALUE`를 그냥 쓰면 "현재 행까지의 마지막 값" = 현재 행의 값이 나옴
- 반드시 `ROWS BETWEEN UNBOUNDED PRECEDING AND UNBOUNDED FOLLOWING`을 명시해야 함
- 이 함정은 SQL 면접 단골 질문!

## 3. 퍼널 분석

### 왜 중요한가?

당근의 핵심 퍼널: 앱 오픈 → 검색/탐색 → 게시글 조회 → 채팅 → 거래 완료 이 퍼널의 각 단계별 전환율을 계산하는 것은 PM/분석가가 가장 많이 요청하는 유형입니다.

### 문제 3-1: 단계별 전환율 계산

> 2026년 3월 한 달간의 중고거래 퍼널 전환율을 구하세요. 퍼널: app_open → view_post → chat_start → transaction_complete

```sql
WITH funnel AS (
    SELECT
        -- Step 1: 앱 오픈 유저
        COUNT(DISTINCT CASE
            WHEN a.activity_type = 'app_open' THEN a.user_id
        END) AS step1_app_open,

        -- Step 2: 게시글 조회 유저
        COUNT(DISTINCT CASE
            WHEN a.activity_type = 'view_post' THEN a.user_id
        END) AS step2_view_post,

        -- Step 3: 채팅 시작 유저 (buyer 기준)
        COUNT(DISTINCT c.buyer_id) AS step3_chat,

        -- Step 4: 거래 완료 유저 (buyer 기준)
        COUNT(DISTINCT t.buyer_id) AS step4_transaction

    FROM user_activity a
    LEFT JOIN chats c
        ON a.user_id = c.buyer_id
        AND c.first_msg_at BETWEEN '2026-03-01' AND '2026-03-31'
    LEFT JOIN transactions t
        ON a.user_id = t.buyer_id
        AND t.completed_at BETWEEN '2026-03-01' AND '2026-03-31'
    WHERE a.activity_date BETWEEN '2026-03-01' AND '2026-03-31'
)

SELECT
    step1_app_open,
    step2_view_post,
    step3_chat,
    step4_transaction,
    ROUND(step2_view_post * 100.0 / NULLIF(step1_app_open, 0), 2) AS cvr_1_to_2,
    ROUND(step3_chat * 100.0 / NULLIF(step2_view_post, 0), 2) AS cvr_2_to_3,
    ROUND(step4_transaction * 100.0 / NULLIF(step3_chat, 0), 2) AS cvr_3_to_4,
    ROUND(step4_transaction * 100.0 / NULLIF(step1_app_open, 0), 2) AS overall_cvr
FROM funnel;
```

### 문제 3-2: 순서를 보장하는 엄격한 퍼널

> 위 퍼널에서 반드시 순서대로 진행한 유저만 카운트하세요. (view_post가 app_open 이후여야 하고, chat이 view_post 이후여야 함)

```sql
WITH user_steps AS (
    SELECT
        user_id,
        MIN(CASE WHEN activity_type = 'app_open'  THEN activity_date END) AS first_open,
        MIN(CASE WHEN activity_type = 'view_post' THEN activity_date END) AS first_view,
        MIN(CASE WHEN activity_type = 'chat'      THEN activity_date END) AS first_chat
    FROM user_activity
    WHERE activity_date BETWEEN '2026-03-01' AND '2026-03-31'
    GROUP BY user_id
)

SELECT
    COUNT(*) AS total_users,
    COUNT(CASE WHEN first_open IS NOT NULL THEN 1 END) AS step1,
    COUNT(CASE WHEN first_view > first_open THEN 1 END) AS step2,
    COUNT(CASE WHEN first_chat > first_view AND first_view > first_open THEN 1 END) AS step3
FROM user_steps;
```

**핵심 포인트:**

- 느슨한 퍼널: 기간 내에 해당 행동을 했는지만 체크 (3-1)
- 엄격한 퍼널: 행동의 순서까지 체크 (3-2)
- 면접에서 "이 퍼널은 순서를 보장하나요?"라고 역질문하면 좋은 인상을 줄 수 있음
- `NULLIF(x, 0)` — 0으로 나누기 방지. 실무에서 필수!

## 4. Self Join을 활용한 연속 이벤트 탐지

### 문제 4-1: 연속 N일 활성 유저 찾기

> 7일 연속으로 앱에 접속한 유저를 찾으세요.

```sql
WITH daily_active AS (
    SELECT DISTINCT user_id, activity_date
    FROM user_activity
),

with_row AS (
    SELECT
        user_id,
        activity_date,
        -- 연속 날짜 그룹을 만드는 트릭:
        -- activity_date에서 row_number를 빼면, 연속된 날짜는 같은 값이 됨
        DATE_SUB(
            activity_date,
            INTERVAL ROW_NUMBER() OVER (
                PARTITION BY user_id ORDER BY activity_date
            ) DAY
        ) AS grp
    FROM daily_active
)

SELECT
    user_id,
    MIN(activity_date) AS streak_start,
    MAX(activity_date) AS streak_end,
    COUNT(*) AS consecutive_days
FROM with_row
GROUP BY user_id, grp
HAVING COUNT(*) >= 7
ORDER BY consecutive_days DESC;
```

**이 트릭의 원리:**

```
activity_date | row_number | date - row_number (= grp)
2026-03-01    | 1          | 2026-02-28
2026-03-02    | 2          | 2026-02-28  ← 같은 grp = 연속!
2026-03-03    | 3          | 2026-02-28  ← 같은 grp = 연속!
2026-03-05    | 4          | 2026-03-01  ← 다른 grp = 끊김
2026-03-06    | 5          | 2026-03-01
```

### 문제 4-2: Self Join 방식 (LAG 활용)

> 3일 연속 게시글을 올린 판매자를 찾으세요.

```sql
WITH daily_posts AS (
    SELECT DISTINCT
        user_id,
        DATE(created_at) AS post_date
    FROM posts
),

with_lag AS (
    SELECT
        user_id,
        post_date,
        LAG(post_date, 1) OVER (PARTITION BY user_id ORDER BY post_date) AS prev_1,
        LAG(post_date, 2) OVER (PARTITION BY user_id ORDER BY post_date) AS prev_2
    FROM daily_posts
)

SELECT DISTINCT user_id
FROM with_lag
WHERE DATE_DIFF(post_date, prev_1, DAY) = 1   -- 어제도 올림
  AND DATE_DIFF(prev_1, prev_2, DAY) = 1;     -- 그저께도 올림
```

**두 가지 접근법 비교:**

|방식|장점|단점|
|---|---|---|
|date - row_number 트릭|N일 연속을 유연하게 처리|트릭을 모르면 떠올리기 어려움|
|LAG/LEAD 비교|직관적, 설명하기 쉬움|N이 커지면 LAG를 여러 번 써야 함|

## 5. GROUP BY + HAVING 조합

### 문제 5-1: 조건부 집계 필터링

> 최근 30일간 게시글을 3개 이상 올렸지만 거래 완료가 한 건도 없는 유저를 찾으세요.

```sql
SELECT
    p.user_id,
    COUNT(DISTINCT p.post_id) AS post_count,
    COUNT(DISTINCT t.transaction_id) AS txn_count
FROM posts p
LEFT JOIN transactions t
    ON p.post_id = t.post_id
WHERE p.created_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 30 DAY)
GROUP BY p.user_id
HAVING COUNT(DISTINCT p.post_id) >= 3
   AND COUNT(DISTINCT t.transaction_id) = 0
ORDER BY post_count DESC;
```

### 문제 5-2: 동네별 거래 활성도 분석

> 동네(region)별 월간 활성 유저 수(MAU), 거래 건수, 유저당 평균 거래 건수를 구하고, MAU 100명 이상인 동네만 필터링하세요.

```sql
WITH monthly_stats AS (
    SELECT
        u.region,
        DATE_TRUNC('month', a.activity_date) AS month,
        COUNT(DISTINCT a.user_id) AS mau,
        COUNT(DISTINCT t.transaction_id) AS txn_count
    FROM users u
    INNER JOIN user_activity a ON u.user_id = a.user_id
    LEFT JOIN transactions t
        ON u.user_id = t.buyer_id
        AND DATE_TRUNC('month', t.completed_at) = DATE_TRUNC('month', a.activity_date)
    WHERE a.activity_date >= '2026-01-01'
    GROUP BY u.region, DATE_TRUNC('month', a.activity_date)
)

SELECT
    region,
    month,
    mau,
    txn_count,
    ROUND(txn_count * 1.0 / mau, 2) AS txn_per_user
FROM monthly_stats
WHERE mau >= 100
ORDER BY month, txn_per_user DESC;
```

**핵심 포인트:**

- `WHERE`는 집계 전 필터, `HAVING`은 집계 후 필터 — 이 차이를 명확히 설명할 것
- `LEFT JOIN` + `COUNT(DISTINCT)` 조합으로 있을 수도/없을 수도 있는 데이터를 안전하게 집계
- `NULLIF`, `COALESCE`로 0 나누기 및 NULL 처리

### 문제 6-1: 중복 데이터 정리

-- 방법 1: ROW_NUMBER로 중복 제거 (가장 일반적)

### 문제 6-2: NULL 안전 처리

> 유저별 마지막 활동일과 현재 날짜의 차이를 구하되, 활동 기록이 없는 유저는 '활동 없음'으로 표시하세요.

```sql
SELECT
    u.user_id,
    u.signup_date,
    MAX(a.activity_date) AS last_active,
    COALESCE(
        CAST(DATE_DIFF(CURRENT_DATE(), MAX(a.activity_date), DAY) AS STRING),
        '활동 없음'
    ) AS days_since_last_active
FROM users u
LEFT JOIN user_activity a ON u.user_id = a.user_id
GROUP BY u.user_id, u.signup_date;
```

**자주 쓰는 NULL 처리 함수:**

|함수|용도|예시|
|---|---|---|
|`COALESCE(a, b, c)`|첫 번째 non-NULL 반환|`COALESCE(phone, email, 'N/A')`|
|`NULLIF(a, b)`|a=b이면 NULL 반환|`x / NULLIF(y, 0)` — 0 나누기 방지|
|`IFNULL(a, b)`|a가 NULL이면 b (BigQuery)|`IFNULL(region, '미설정')`|
|`NVL(a, b)`|IFNULL과 동일 (Redshift/Oracle)|`NVL(region, '미설정')`|

### 문제 6-3: 날짜 연산 종합

> 가입 후 7일 이내에 첫 거래를 완료한 유저의 비율을 월별로 구하세요.

```sql
WITH first_txn AS (
    SELECT
        buyer_id AS user_id,
        MIN(completed_at) AS first_txn_date
    FROM transactions
    GROUP BY buyer_id
)

SELECT
    DATE_TRUNC('month', u.signup_date) AS signup_month,
    COUNT(DISTINCT u.user_id) AS total_signups,
    COUNT(DISTINCT CASE
        WHEN DATE_DIFF(ft.first_txn_date, u.signup_date, DAY) <= 7
        THEN u.user_id
    END) AS converted_in_7d,
    ROUND(
        COUNT(DISTINCT CASE
            WHEN DATE_DIFF(ft.first_txn_date, u.signup_date, DAY) <= 7
            THEN u.user_id
        END) * 100.0 / COUNT(DISTINCT u.user_id),
        2
    ) AS conversion_rate_7d
FROM users u
LEFT JOIN first_txn ft ON u.user_id = ft.user_id
GROUP BY DATE_TRUNC('month', u.signup_date)
ORDER BY signup_month;
```

**날짜 함수 정리 (BigQuery vs Redshift):**

|목적|BigQuery|Redshift|
|---|---|---|
|날짜 차이|`DATE_DIFF(end, start, DAY)`|`DATEDIFF(day, start, end)`|
|날짜 더하기|`DATE_ADD(date, INTERVAL 7 DAY)`|`DATEADD(day, 7, date)`|
|월 시작일|`DATE_TRUNC('month', date)`|`DATE_TRUNC('month', date)`|
|현재 날짜|`CURRENT_DATE()`|`CURRENT_DATE` (괄호 없음)|
|타임스탬프→날짜|`DATE(timestamp)`|`timestamp::date`|

## 7. CTE vs 서브쿼리

### CTE (Common Table Expression) 권장 패턴

실무에서는 CTE가 압도적으로 선호됩니다. dbt도 CTE 기반이에요.

```sql
-- ✅ CTE: 읽기 쉽고, 디버깅 쉽고, 재사용 가능
WITH daily_users AS (
    SELECT activity_date, COUNT(DISTINCT user_id) AS dau
    FROM user_activity
    GROUP BY activity_date
),
weekly_avg AS (
    SELECT
        DATE_TRUNC('week', activity_date) AS week,
        ROUND(AVG(dau), 0) AS avg_dau
    FROM daily_users
    GROUP BY DATE_TRUNC('week', activity_date)
)
SELECT * FROM weekly_avg ORDER BY week;


-- ❌ 서브쿼리 중첩: 같은 로직이지만 읽기 어려움
SELECT
    DATE_TRUNC('week', activity_date) AS week,
    ROUND(AVG(dau), 0) AS avg_dau
FROM (
    SELECT activity_date, COUNT(DISTINCT user_id) AS dau
    FROM user_activity
    GROUP BY activity_date
) daily_users
GROUP BY DATE_TRUNC('week', activity_date)
ORDER BY week;
```

**면접에서 CTE를 쓰면 좋은 이유:**

- 사고 과정이 위에서 아래로 자연스럽게 흐름 → 면접관이 따라가기 쉬움
- 중간 결과를 여러 번 참조 가능 (서브쿼리는 매번 복붙해야 함)
- dbt 모델 구조와 동일하므로 당근 스택과 친화적

**서브쿼리가 더 나은 경우:**

- `WHERE x IN (SELECT ...)` 같은 단순 필터
- `EXISTS` / `NOT EXISTS` 패턴

```sql
-- EXISTS가 적절한 경우: 게시글은 있지만 채팅이 한 건도 없는 게시글
SELECT p.*
FROM posts p
WHERE NOT EXISTS (
    SELECT 1 FROM chats c WHERE c.post_id = p.post_id
);
```

## 8. 잘 안 쓰지만 알아둬야 할 함수들

실무에서 자주 쓰진 않지만, 면접에서 나오면 "이 함수도 아세요?" 가산점을 받을 수 있는 함수들입니다.

### 8-1. 문자열 집계 — STRING_AGG / LISTAGG / ARRAY_AGG

> 동네별로 등록된 게시글 카테고리 목록을 콤마로 구분해서 보여주세요.

```sql
-- BigQuery
SELECT
    u.region,
    STRING_AGG(DISTINCT p.category, ', ' ORDER BY p.category) AS categories
FROM users u
JOIN posts p ON u.user_id = p.user_id
GROUP BY u.region;

-- Redshift / PostgreSQL
SELECT
    u.region,
    LISTAGG(DISTINCT p.category, ', ') WITHIN GROUP (ORDER BY p.category) AS categories
FROM users u
JOIN posts p ON u.user_id = p.user_id
GROUP BY u.region;

-- 배열로 받고 싶으면
SELECT
    u.region,
    ARRAY_AGG(DISTINCT p.category ORDER BY p.category) AS category_array
FROM users u
JOIN posts p ON u.user_id = p.user_id
GROUP BY u.region;
```

|DB|함수|
|---|---|
|BigQuery|`STRING_AGG(col, sep)`|
|Redshift / PostgreSQL|`LISTAGG(col, sep) WITHIN GROUP (ORDER BY ...)`|
|MySQL|`GROUP_CONCAT(col SEPARATOR sep)`|
|Snowflake|`LISTAGG` 또는 `ARRAY_AGG`|

- **자주 쓰는 곳**: 디버깅용 그룹별 ID 모아보기, 한 줄에 여러 값 펼쳐서 보고하기
- **주의**: 결과 길이 제한이 있는 DB가 많음 (Redshift LISTAGG는 65535 바이트)

### 8-2. 백분위수 — PERCENTILE_CONT / APPROX_QUANTILES

> 카테고리별 거래 금액의 중앙값(median), 25%/75% 분위수, 90% 분위수를 구하세요.

```sql
-- PostgreSQL / Redshift
SELECT
    p.category,
    PERCENTILE_CONT(0.25) WITHIN GROUP (ORDER BY t.amount) AS q1,
    PERCENTILE_CONT(0.50) WITHIN GROUP (ORDER BY t.amount) AS median,
    PERCENTILE_CONT(0.75) WITHIN GROUP (ORDER BY t.amount) AS q3,
    PERCENTILE_CONT(0.90) WITHIN GROUP (ORDER BY t.amount) AS p90
FROM transactions t
JOIN posts p ON t.post_id = p.post_id
GROUP BY p.category;

-- BigQuery (정확)
SELECT
    p.category,
    PERCENTILE_CONT(t.amount, 0.5) OVER (PARTITION BY p.category) AS median
FROM transactions t
JOIN posts p ON t.post_id = p.post_id;

-- BigQuery (근사 — 빠름, 대용량에 적합)
SELECT
    p.category,
    APPROX_QUANTILES(t.amount, 100)[OFFSET(50)] AS approx_median,
    APPROX_QUANTILES(t.amount, 100)[OFFSET(90)] AS approx_p90
FROM transactions t
JOIN posts p ON t.post_id = p.post_id
GROUP BY p.category;
```

**왜 평균 대신 중앙값을 쓸까?**

- 거래 금액처럼 분포가 한쪽으로 치우친 데이터(롱테일)는 평균이 왜곡됨
- 면접 팁: "이 지표는 평균보다 중앙값이 더 적합합니다. 분포가 right-skewed이기 때문입니다" 같은 멘트 가능
- `PERCENTILE_CONT` (연속) vs `PERCENTILE_DISC` (이산) — 작은 데이터셋에선 차이가 큼

### 8-3. 피벗(PIVOT) — CASE WHEN으로 행→열 변환

> 월별, 카테고리별 거래 건수를 카테고리가 컬럼이 되도록 피벗하세요.

```sql
-- 표준 SQL: CASE WHEN으로 수동 피벗 (가장 호환성 좋음)
SELECT
    DATE_TRUNC('month', t.completed_at) AS month,
    COUNT(CASE WHEN p.category = '디지털기기' THEN 1 END) AS digital,
    COUNT(CASE WHEN p.category = '가구' THEN 1 END) AS furniture,
    COUNT(CASE WHEN p.category = '의류' THEN 1 END) AS clothing,
    COUNT(*) AS total
FROM transactions t
JOIN posts p ON t.post_id = p.post_id
GROUP BY DATE_TRUNC('month', t.completed_at)
ORDER BY month;

-- BigQuery: PIVOT 구문 (카테고리가 정해져 있을 때)
SELECT *
FROM (
    SELECT
        DATE_TRUNC(t.completed_at, MONTH) AS month,
        p.category,
        t.transaction_id
    FROM transactions t
    JOIN posts p ON t.post_id = p.post_id
)
PIVOT (
    COUNT(transaction_id)
    FOR category IN ('디지털기기', '가구', '의류')
);
```

- **언피벗(UNPIVOT)** 도 있음 — 컬럼 → 행 변환. BigQuery/Snowflake/Redshift에서 지원
- 면접 팁: "PIVOT 구문을 모르더라도 CASE WHEN으로 풀 수 있다"는 점을 강조하면 좋음

### 8-4. 그룹 집계 확장 — GROUPING SETS / ROLLUP / CUBE

> 동네별, 카테고리별, 동네+카테고리별 거래 건수를 하나의 쿼리로 모두 구하세요.

```sql
SELECT
    u.region,
    p.category,
    COUNT(*) AS txn_count,
    GROUPING(u.region)   AS is_region_total,
    GROUPING(p.category) AS is_category_total
FROM transactions t
JOIN users u ON t.buyer_id = u.user_id
JOIN posts p ON t.post_id = p.post_id
GROUP BY GROUPING SETS (
    (u.region, p.category),  -- 동네+카테고리별
    (u.region),              -- 동네별 소계
    (p.category),            -- 카테고리별 소계
    ()                       -- 전체 합계
);
```

|구문|의미|
|---|---|
|`GROUPING SETS ((a,b), (a), ())`|명시한 조합만 집계|
|`ROLLUP(a, b)`|(a,b), (a), () — 계층적 소계|
|`CUBE(a, b)`|(a,b), (a), (b), () — 모든 조합|

- 대시보드용 보고서 쿼리에서 자주 사용 (1개 쿼리로 소계+합계 다 뽑기)
- `GROUPING(col)` = 1이면 해당 컬럼이 집계된 소계 행임을 나타냄

### 8-5. QUALIFY — 윈도우 함수 결과 직접 필터링

> 카테고리별 거래액 1위 게시글만 뽑으세요.

```sql
-- 일반적인 방식 (CTE + WHERE)
WITH ranked AS (
    SELECT
        post_id, category, amount,
        ROW_NUMBER() OVER (PARTITION BY category ORDER BY amount DESC) AS rn
    FROM transactions t JOIN posts p ON t.post_id = p.post_id
)
SELECT * FROM ranked WHERE rn = 1;

-- QUALIFY (BigQuery / Snowflake / Databricks) — 1단계로 끝
SELECT
    t.post_id, p.category, t.amount
FROM transactions t
JOIN posts p ON t.post_id = p.post_id
QUALIFY ROW_NUMBER() OVER (PARTITION BY p.category ORDER BY t.amount DESC) = 1;
```

- `QUALIFY`는 윈도우 함수 결과에 `WHERE`를 거는 것과 같음
- Redshift는 미지원 (CTE로 우회 필요)

### 8-6. LATERAL / UNNEST — 배열 펼치기

> activity_type이 배열로 저장된 가상의 테이블에서 각 활동을 한 행씩 펼치세요.

```sql
-- 예시 데이터: user_sessions(user_id, session_id, activities ARRAY<STRING>)

-- BigQuery (UNNEST)
SELECT
    s.user_id,
    s.session_id,
    activity
FROM user_sessions s,
UNNEST(s.activities) AS activity;

-- PostgreSQL (LATERAL)
SELECT
    s.user_id,
    s.session_id,
    activity
FROM user_sessions s,
LATERAL UNNEST(s.activities) AS activity;
```

- 로그 데이터(JSON 배열, 이벤트 리스트)를 다룰 때 필수
- BigQuery/Snowflake에서는 거의 일상적으로 사용

### 8-7. 정규식 — REGEXP_EXTRACT / REGEXP_REPLACE

> 게시글 제목에서 가격 표기(예: "10만원", "5,000원")를 추출하세요.

-- BigQuery
SELECT
    post_id,

    REGEXP_EXTRACT(title, r'(\d{1,3}(,\d{3})*|\d+)\s*원') AS extracted_price,

SELECT

### 8-8. JSON 함수

> 이벤트 로그의 JSON payload에서 referrer 값을 추출하세요. (event_log(payload STRING) 가정)

-- BigQuery
SELECT

### 8-9. 조건부 평균 — FILTER 절

> 전체 거래 건수, 5만원 이상 거래 건수, 5만원 이상 거래의 평균 금액을 한 번에 구하세요.

```sql
-- PostgreSQL / Redshift / BigQuery
SELECT
    COUNT(*) AS total_txn,
    COUNT(*) FILTER (WHERE amount >= 50000) AS high_value_txn,
    AVG(amount) FILTER (WHERE amount >= 50000) AS avg_high_value
FROM transactions;

-- 표준 호환 방식 (CASE WHEN)
SELECT
    COUNT(*) AS total_txn,
    COUNT(CASE WHEN amount >= 50000 THEN 1 END) AS high_value_txn,
    AVG(CASE WHEN amount >= 50000 THEN amount END) AS avg_high_value
FROM transactions;
```

- `FILTER` 절이 더 가독성 좋음 (PostgreSQL/Redshift 지원, BigQuery는 미지원)
- 한 번의 스캔으로 여러 조건의 집계를 동시에 — 성능 친화적

### 8-10. INTERSECT / EXCEPT — 집합 연산

> 게시글을 올린 적도 있고 거래도 한 유저 vs 게시글만 올리고 거래는 안 한 유저

```sql
-- 둘 다 있는 유저 (교집합)
SELECT user_id FROM posts
INTERSECT
SELECT seller_id FROM transactions;

-- 게시글은 올렸지만 거래는 없는 유저 (차집합)
SELECT user_id FROM posts
EXCEPT
SELECT seller_id FROM transactions;
```

- `EXCEPT`는 MySQL에선 `MINUS`로 표기 (Oracle도 마찬가지)
- 중복을 유지하고 싶으면 `INTERSECT ALL`, `EXCEPT ALL`
- `NOT EXISTS`로도 같은 결과를 낼 수 있지만 의도가 더 명확함

## 9. 종합 문제 (실전 시뮬레이션)

### 문제 9-1: A/B 테스트 결과 분석

> 검색 알고리즘 A/B 테스트를 진행했습니다. 실험 그룹(experiment)과 대조 그룹(control)의 전환율(검색 → 게시글 조회 → 채팅)을 비교하세요.

```sql
-- 실험 그룹 배정 테이블이 있다고 가정
-- ab_test_groups (user_id, test_name, group_name, assigned_at)

WITH search_users AS (
    SELECT
        g.group_name,
        a.user_id,
        MIN(a.activity_date) AS first_search_date
    FROM ab_test_groups g
    INNER JOIN user_activity a
        ON g.user_id = a.user_id
        AND a.activity_type = 'search'
        AND a.activity_date >= g.assigned_at
    WHERE g.test_name = 'search_algo_v2'
    GROUP BY g.group_name, a.user_id
),

funnel AS (
    SELECT
        su.group_name,
        su.user_id,
        -- 검색 후 게시글 조회 여부
        MAX(CASE WHEN va.activity_type = 'view_post'
                  AND va.activity_date >= su.first_search_date
             THEN 1 ELSE 0 END) AS viewed_post,
        -- 검색 후 채팅 여부
        MAX(CASE WHEN c.buyer_id IS NOT NULL THEN 1 ELSE 0 END) AS started_chat
    FROM search_users su
    LEFT JOIN user_activity va
        ON su.user_id = va.user_id
        AND va.activity_date >= su.first_search_date
    LEFT JOIN chats c
        ON su.user_id = c.buyer_id
        AND c.first_msg_at >= su.first_search_date
    GROUP BY su.group_name, su.user_id
)

SELECT
    group_name,
    COUNT(*) AS searchers,
    SUM(viewed_post) AS viewers,
    SUM(started_chat) AS chatters,
    ROUND(SUM(viewed_post) * 100.0 / COUNT(*), 2) AS search_to_view_rate,
    ROUND(SUM(started_chat) * 100.0 / NULLIF(SUM(viewed_post), 0), 2) AS view_to_chat_rate,
    ROUND(SUM(started_chat) * 100.0 / COUNT(*), 2) AS overall_cvr
FROM funnel
GROUP BY group_name;
```

### 문제 9-2: 파워 셀러 분석

> 아래 조건을 모두 만족하는 '파워 셀러'를 찾고, 동네별 파워 셀러 비율을 구하세요. 조건: ① 최근 90일 거래 완료 5건 이상, ② 평균 판매가 5만원 이상, ③ 채팅 응답 게시글 비율 80% 이상

```sql
WITH seller_metrics AS (
    SELECT
        t.seller_id,
        COUNT(DISTINCT t.transaction_id) AS txn_count,
        AVG(t.amount) AS avg_price,
        COUNT(DISTINCT t.post_id) AS sold_posts
    FROM transactions t
    WHERE t.completed_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
    GROUP BY t.seller_id
),

chat_response AS (
    SELECT
        p.user_id AS seller_id,
        COUNT(DISTINCT p.post_id) AS total_posts,
        COUNT(DISTINCT c.post_id) AS posts_with_chat
    FROM posts p
    LEFT JOIN chats c ON p.post_id = c.post_id
    WHERE p.created_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
      AND p.status != 'deleted'
    GROUP BY p.user_id
),

power_sellers AS (
    SELECT sm.seller_id
    FROM seller_metrics sm
    INNER JOIN chat_response cr ON sm.seller_id = cr.seller_id
    WHERE sm.txn_count >= 5
      AND sm.avg_price >= 50000
      AND cr.posts_with_chat * 1.0 / NULLIF(cr.total_posts, 0) >= 0.8
)

SELECT
    u.region,
    COUNT(DISTINCT u.user_id) AS total_sellers,
    COUNT(DISTINCT ps.seller_id) AS power_seller_count,
    ROUND(COUNT(DISTINCT ps.seller_id) * 100.0
        / NULLIF(COUNT(DISTINCT u.user_id), 0), 2) AS power_seller_rate
FROM users u
LEFT JOIN power_sellers ps ON u.user_id = ps.seller_id
-- 게시글을 하나라도 올린 유저만 셀러로 간주
WHERE EXISTS (SELECT 1 FROM posts p WHERE p.user_id = u.user_id)
GROUP BY u.region
ORDER BY power_seller_rate DESC;
```

### 문제 9-3: RFM 세그먼트 (백분위수 + NTILE 종합)

> 최근 90일 데이터로 구매자를 RFM(Recency, Frequency, Monetary) 5분위로 점수화하고, R+F+M 합계가 12점 이상인 'VIP' 유저를 찾으세요.

```sql
WITH base AS (
    SELECT
        buyer_id,
        DATE_DIFF(CURRENT_DATE(), MAX(DATE(completed_at)), DAY) AS recency_days,
        COUNT(*) AS frequency,
        SUM(amount) AS monetary
    FROM transactions
    WHERE completed_at >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)
    GROUP BY buyer_id
),

scored AS (
    SELECT
        buyer_id,
        recency_days,
        frequency,
        monetary,
        -- Recency: 최근에 구매할수록 높은 점수 (역순)
        NTILE(5) OVER (ORDER BY recency_days DESC) AS r_score,
        -- Frequency: 자주 구매할수록 높은 점수
        NTILE(5) OVER (ORDER BY frequency ASC) AS f_score,
        -- Monetary: 많이 구매할수록 높은 점수
        NTILE(5) OVER (ORDER BY monetary ASC) AS m_score
    FROM base
)

SELECT
    buyer_id,
    recency_days,
    frequency,
    monetary,
    r_score, f_score, m_score,
    (r_score + f_score + m_score) AS rfm_total,
    CASE
        WHEN (r_score + f_score + m_score) >= 13 THEN 'VVIP'
        WHEN (r_score + f_score + m_score) >= 11 THEN 'VIP'
        WHEN (r_score + f_score + m_score) >= 8  THEN 'Active'
        WHEN r_score <= 2                         THEN 'Churn Risk'
        ELSE 'Regular'
    END AS segment
FROM scored
WHERE (r_score + f_score + m_score) >= 12
ORDER BY rfm_total DESC, monetary DESC;
```

**핵심 포인트:**

- `NTILE`로 점수화하는 것이 RFM의 정석
- Recency는 "작을수록 좋음"이라 `ORDER BY recency_days DESC`로 역정렬
- 면접 팁: "비즈니스 정의가 5분위가 아닌 4분위라면 NTILE(4)로 바꾸기만 하면 됩니다" — 유연성을 강조하면 좋음

면접 중 문제를 받았을 때 순서:

1. **문제 이해 확인** — "이 문제에서 리텐션은 Classic인가요 Rolling인가요?" 같은 질문
2. **가정 명시** — "user_activity에 중복 행이 있을 수 있다고 가정하고 DISTINCT 처리하겠습니다"
3. **접근 방법 설명** — "CTE 3단계로 구성하겠습니다. 첫 번째는 ~, 두 번째는 ~"
4. **코드 작성** — CTE 이름을 의미 있게 지을 것 (daily_users, ranked_txn 등)
5. **엣지 케이스 언급** — "0으로 나누는 경우를 NULLIF로 처리했습니다"
6. **성능 고려** — "실무에서는 파티션 프루닝을 위해 날짜 필터를 먼저 걸겠습니다"

### 자주 실수하는 것들

- `COUNT(DISTINCT)` 빠뜨림 → 중복 카운트
- `LEFT JOIN` 해야 하는데 `INNER JOIN` 사용 → 데이터 누락
- Window Function에서 `PARTITION BY` 빠뜨림 → 전체 대상으로 계산
- `GROUP BY`에 SELECT의 비집계 컬럼 빠뜨림 → 에러
- 날짜 비교에서 타임스탬프 vs DATE 타입 불일치 → 의도치 않은 결과
- `LAST_VALUE()`에서 프레임 명시 안 함 → 현재 행 값이 나옴
- `WHERE` 절에 윈도우 함수 사용 시도 → `QUALIFY`나 CTE로 우회 필요
- 정수 나눗셈으로 비율이 0이 됨 → `* 100.0` 또는 `CAST(... AS FLOAT)` 사용
- `HAVING`에서 alias 못 쓰는 DB 있음 (PostgreSQL은 가능, 일부 안 됨) → 표현식 그대로 작성
- `NULL = NULL`은 `FALSE` (TRUE 아님!) → `IS NULL` 또는 `IS NOT DISTINCT FROM` 사용
